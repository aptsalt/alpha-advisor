ALPHA Advisor, explained

A multi-agent copilot that prepares a wealth-management client portfolio review — under human supervision. It does the work autonomously, but pauses for the advisor to approve before anything reaches the client, and logs every step to a tamper-proof trail.

The key idea: the orchestration is the product. The LLM is a swappable component — what makes this deployable in a regulated bank is the graph of steps, the guardrails, and the audit trail around the LLM.

00The mental model — one request

Type "Prepare a portfolio review for Jane Harrington" and the request flows through a pipeline of specialized steps (a LangGraph state machine). Each step does one job and hands a shared state object to the next.

plan→ input_guard→ retrieve ⇄ rewrite→ market→ compliance→ rebalance→ draft→ output_guard→ approval ⏸→ finalize

In one breath: it figures out what you're asking → scrubs sensitive data → gathers evidence → checks prices → runs compliance → proposes fixes → writes a cited briefing → double-checks it → stops and waits for you to approve → finalizes. (Click any step above to jump to it.)

01The pieces, in plain terms

Each node is a small, single-responsibility function: it takes the shared state and returns a partial update. That's why the run reads like the advisory workflow itself.

1

plan nodes/plan.py reason

Classifies intent ("portfolio review" vs "lookup"), resolves which client you mean ("Jane" → C-1001), and lays out the sub-tasks. This is the reasoning half of an agent loop.

2

input_guard nodes/guardrails.py governance · in

Governance before the model. Redacts PII (the model never needs Jane's SSN to reason about allocation — showing it less is itself a control) and hard-blocks prohibited requests (e.g. "guarantee 20% returns"). A blocked request routes straight to refuse and never touches client data.

3

retrieve ⇄ rewrite nodes/retrieve.py agentic RAG ★

The centerpiece. Plain RAG: embed query → grab top docs → stuff into the prompt — one shot, no judgment. Agentic RAG makes retrieval a decision: it pulls from two sources — a vector store (prose like policies) and the knowledge graph (the client's portfolio structure) — grades each document for relevance, and if the evidence is weak, rewrites the query and retries. That self-correction is the difference between a plausible and a defensible answer in a bank.

4

market nodes/market.py tool

Calls a market-data tool for prices and returns. Key point: the numbers come from a tool, not from the model making them up.

5

compliance nodes/compliance.py · advice.py governance

Three checks, each a graph query with a threshold: suitability (does the allocation match the risk profile?), restricted-list (does any holding touch a banned security?), concentration (is any single issuer over the 25% limit?). Each returns pass / warn / fail with a plain-English reason.

6

rebalance nodes/rebalance.py advice

If compliance found problems, it computes specific trades to fix them — sell the restricted holding, trim the over-concentrated issuer, reallocate to a diversified core. The math is computed deterministically, never generated. This turns the app from "review" into "advise".

7

draft nodes/draft.py grounding

Writes the client briefing from all the gathered evidence, citing every factual claim [1][2]…, and streams out token-by-token live.

8

output_guard nodes/guardrails.py governance · out

Governance after drafting: rejects an uncited draft, scrubs any leaked PII, and appends the "not investment advice until approved" disclaimer.

9

approval ⏸ nodes/approval.py human-in-the-loop

The human-in-the-loop. A real LangGraph interrupt(): the graph pauses, saves its entire state, and surfaces the briefing to the advisor. Nothing finalizes until you click Allow or Reject. The advisor is always the last actor before the client.

10

finalize / discard nodes/finalize.py terminal

On approval, finalize and write the closing audit record. On rejection, discard with the reason.

Running alongside the whole thing: audit.py writes one append-only, hash-chained record per step (each line's hash includes the previous line's, so any tampering is detectable), and telemetry.py emits an OpenTelemetry span per node.

03The three concepts they'll probe

Click to reveal the crisp answer. Each is anchored to something you actually built.

Naive RAG vs Agentic RAG? +
Naive = embed → top-k → generate. One shot, no self-check. Agentic = retrieval is a decision: it grades its own results and rewrites/retries on weak evidence. In wealth management that's the line between plausible and defensible.
→ nodes/retrieve.py · the should_retry conditional edge
Why a knowledge graph and not just vectors? +
Vectors answer "what text is similar." A graph answers structural, multi-hop questions. My demo catches a 78% issuer concentration by aggregating two different funds (GLX + AGG) that share one issuer (Vantage) — a graph traversal vectors physically cannot do.
→ rag/graphrag.py · issuer_concentration()
How is an autonomous agent safe to deploy in a bank? +
Three things: guardrails on both ends (PII redaction in; citation / PII / disclaimer checks out), a real human approval gate before anything reaches a client, and a tamper-evident, hash-chained audit trail. Plus self-evaluation (an LLM-as-judge scores groundedness and suitability). Autonomy with a human backstop, not instead of one.
→ guardrails.py · approval.py · audit.py · evaluate.py
Why LangGraph over AutoGen / CrewAI / Semantic Kernel? +
The workflow is a stateful graph with conditional loops and a durable human pause — LangGraph's exact model, giving deterministic, inspectable, resumable control that a regulated setting needs. Semantic Kernel is the natural pick if the bank is Azure/.NET end-to-end; the nodes port to plugins, the graph shape and guardrails don't change. I keep the orchestration contract fixed so the framework is swappable.

04The choices that make it "senior"

These matter more than any single line of code — they're what an interviewer listens for.

One typed state object. Every node receives the same AdvisorState and returns a partial update. Because all state is explicit and typed, the run is inspectable and the audit trail is a byproduct, not extra work. In a regulated setting, explainability is the architecture.
Provider-portable. mock (no keys) / ollama (local) / azure (production) is a config switch — the graph, guardrails, and audit logic are identical. The same system can run fully on-prem, which matters when client data can't leave the building.
Task-based model routing. A small fast model (qwen2.5:3b) does the high-frequency grading; a stronger one (gemma3:12b) does the drafting. Real cost/latency engineering — you don't pay a 12B model to score a document yes/no.
Stateless graph + durable checkpointer. The graph holds no state between requests; paused runs live in the checkpointer. With Postgres, any replica can resume any run — so it scales horizontally and a run can wait hours for an advisor. That's the cloud-scaling story.

05The 5 "platform" features

Beyond the core review — what turns a tool into a platform.

⚖ Eval harness

LLM-as-judge scores groundedness + suitability + a deterministic citation-coverage metric. "How do you know the agent is right?" → it scores itself, and the score is auditable. (evaluate.py, run_eval.py)

⇄ Rebalancing

Proposed, computed trades to bring the book back within policy. (nodes/rebalance.py)

⌁ Token streaming

The briefing types out live, word by word, over SSE.

◫ Knowledge-graph viz

Interactive client→holding→issuer→sector diagram — restricted = red, over-limit = amber.

▤ Portfolio dashboard

A fast, LLM-free book-wide compliance scan = the "flagged-for-review queue." Pure graph queries, so it's instant across the whole book.

07Quiz yourself

Answer out loud, then reveal. If you can do all six cleanly, you can carry the interview.

1. Walk me through what happens when an advisor asks for a portfolio review.
plan → input_guard (PII + policy) → retrieve⇄rewrite (agentic RAG over vectors + graph) → market tool → compliance (3 graph checks) → rebalance (proposed trades) → draft (cited, streamed) → output_guard → approval interrupt → finalize. Every step writes a hash-chained audit record.
2. What makes your retrieval "agentic"?
It grades each retrieved document for relevance and, on weak evidence, rewrites the query and retries (bounded) — instead of confidently answering from whatever it first pulled.
3. Give one concrete thing the graph finds that vectors can't.
A 78% single-issuer concentration aggregated across two different funds (GLX + AGG) that share one issuer (Vantage). It's a multi-hop relationship query, not a similarity match.
4. How does the human-in-the-loop actually work, technically?
LangGraph interrupt() persists the full graph state to a checkpointer and surfaces the briefing; I resume with Command(resume={...}), and a conditional edge routes to finalize or discard. With Postgres the paused run survives restarts and any replica can resume it.
5. How would you prevent the agent from hallucinating a number?
Market figures come from a tool, rebalance math is computed deterministically, the draft only cites provided evidence, and the output guard rejects an uncited draft. The eval harness then scores groundedness as a backstop.
6. How does this scale on Azure?
Stateless graph + durable (Postgres) checkpointer → containerize → Azure Container Apps, Azure OpenAI, Azure AI Search + Neo4j, OpenTelemetry → Azure Monitor, audit → immutable storage. Horizontal, because any worker resumes any paused run.
